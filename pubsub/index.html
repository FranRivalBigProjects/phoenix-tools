// How Phoenix.PubSub works

Phoenix PubSub — Elixir Publish-Subscribe Pattern

Phoenix.PubSub is the distributed message routing layer used by Phoenix Channels and LiveView. It dispatches messages from a publisher to all processes subscribed to a named topic. The classic usage pattern:

# 1. Start PubSub in your application supervisor children = [ {Phoenix.PubSub, name: MyApp.PubSub} ] # 2. Subscribe a process to a topic Phoenix.PubSub.subscribe(MyApp.PubSub, “room:lobby”) # 3. Broadcast to all subscribers Phoenix.PubSub.broadcast(MyApp.PubSub, “room:lobby”, %{ event: “user_joined”, user: “alice” }) # 4. Receive in subscriber’s handle_info def handle_info(%{event: “user_joined”, user: user}, state) do IO.puts(”#{user} joined”) {:noreply, state} end

What is the difference between broadcast/3 and broadcast_from/4?
broadcast/3 sends a message to all subscribers of a topic including the calling process if it is subscribed. broadcast_from/4 takes an explicit PID as the second argument and excludes that PID from receiving the message. In Phoenix Channels, broadcast_from/4 is used to avoid echoing a message back to the user who sent it — the socket PID is excluded.

How does Phoenix.PubSub handle multi-node clusters?
The default PubSub adapter (Phoenix.PubSub.PG2) uses Erlang process groups which are automatically distributed across all connected BEAM nodes. A broadcast on any node reaches subscribers on all nodes in the cluster. For non-BEAM clusters or when you need cross-datacenter delivery, Phoenix.PubSub.Redis uses Redis pub/sub as the transport layer. Configure the adapter in your application config.

What happens if a subscriber process crashes?
When a subscribed process terminates, Phoenix.PubSub automatically removes it from the topic's subscriber list. There is no manual cleanup required. The PubSub system monitors subscriber PIDs and handles the unsubscription on process exit. This is one of the key advantages of building on the BEAM — process lifecycle management is handled by the runtime.

How does Phoenix LiveView use PubSub?
LiveView sockets subscribe to PubSub topics in mount/3 using Phoenix.PubSub.subscribe/2 or the Phoenix.LiveView.subscribe/2 helper. When a broadcast arrives, LiveView receives it in handle_info/2 and can update assigns to trigger a re-render. This is the standard pattern for real-time collaborative features: one user's action broadcasts to a topic, all connected LiveView processes receive and re-render.

Frequently Asked Questions — Phoenix PubSub

Phoenix.PubSub is the distributed publish-subscribe system in Phoenix Framework. Processes subscribe to named string topics and receive any message broadcast to those topics. It is used internally by Phoenix Channels and LiveView, and can be used directly for any inter-process messaging in Elixir applications.
In init/1, call Phoenix.PubSub.subscribe(MyApp.PubSub, "my:topic"). Then implement handle_info/2 to handle incoming messages. The subscribed GenServer receives broadcasts as standard Elixir messages in its mailbox, processed by handle_info like any other message.
The pubsub pattern in Elixir follows the same principle as other languages but benefits from the BEAM's lightweight processes. Publishers and subscribers are independent processes with no shared state. Topics are strings — any process can subscribe to any topic. The BEAM's message-passing model makes pubsub a natural fit: messages are immutable, delivery is asynchronous, and crashed subscribers are automatically cleaned up.
