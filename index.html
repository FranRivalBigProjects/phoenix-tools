Phoenix Telemetry Systems

Phoenix telemetry systems are built on the Erlang :telemetry library — a lightweight, zero-dependency event dispatching mechanism. Phoenix emits telemetry events at every major lifecycle point: HTTP request start and stop, LiveView mount, socket connect, Ecto query execution, and channel joins. You attach handlers to these events to collect metrics, write structured logs, or send traces to observability backends like Prometheus, Datadog, or OpenTelemetry.

01

Attach a telemetry handler

Call :telemetry.attach/4 with an event name, handler function, and optional config. Phoenix typically does this in your Telemetry.ex supervisor module generated by mix phx.new.

02

Use :telemetry_metrics for aggregation

The :telemetry_metrics library defines metric types — counter, sum, last_value, distribution — that reporters like TelemetryMetricsPrometheus consume to expose a /metrics endpoint.

03

Built-in Phoenix events require no setup

Phoenix emits [:phoenix,:endpoint,:start], [:phoenix,:endpoint,:stop], [:phoenix,:live_view,:mount,:start] and more automatically. Your handlers receive these without any additional instrumentation code.

04

Emit custom events with :telemetry.execute/3

Call :telemetry.execute([:my_app,:payment,:processed], %{duration: elapsed}, %{user_id: user_id}) to emit domain events. Any process with a matching handler attached receives the measurement and metadata maps.

How does Phoenix telemetry differ from Logger?
Logger writes human-readable text to a log sink. :telemetry dispatches structured numeric measurements to attached handler functions. Telemetry is designed for metrics and tracing, not logging — although a handler can write to Logger. The key difference is that telemetry events are synchronous calls into handler functions that run in the emitting process, so handlers must be fast and non-blocking.

What is the Telemetry.ex module generated by Phoenix?
When you run mix phx.new, Phoenix generates a lib/my_app_web/telemetry.ex supervisor module. This module starts telemetry reporter processes (like TelemetryMetricsPrometheus), defines the list of metrics to track using :telemetry_metrics functions, and attaches the handler that Phoenix uses to emit default request metrics. It is the central configuration point for your application's observability.

How do I integrate Phoenix telemetry with OpenTelemetry?
Add opentelemetry_phoenix to your dependencies. It provides a OpentelemetryPhoenix.setup/1 call that attaches handlers to Phoenix telemetry events and converts them to OpenTelemetry spans automatically. Ecto integration is provided by opentelemetry_ecto. This gives you distributed traces across HTTP requests, LiveView events, and database queries with no manual instrumentation.

Frequently Asked Questions
Phoenix telemetry systems refer to the observability infrastructure built into Phoenix using the :telemetry Erlang library. Phoenix emits structured events at key lifecycle points (requests, LiveView mounts, channel joins) that handlers attach to for metrics collection, structured logging, and distributed tracing. The generated Telemetry.ex module in a Phoenix app configures these handlers.
Phoenix.PubSub is a distributed publish-subscribe system. Processes call Phoenix.PubSub.subscribe/2 to receive messages on a topic and Phoenix.PubSub.broadcast/3 to send to all subscribers. Internally it uses a backend (PG2 for single-node, Redis adapter for multi-node clusters). Phoenix Channels and LiveView use PubSub internally for message routing.
Phoenix Channels are the WebSocket abstraction that manages client connections — handling join, leave, and message callbacks. PubSub is the lower-level message routing system that Channels use internally. Use Channels when you need a WebSocket connection with client-facing API. Use PubSub directly for server-to-server or process-to-process communication without a WebSocket connection.
Call :telemetry.attach("my-handler", [:phoenix,:endpoint,:stop], &MyHandler.handle/4, nil) during application startup, typically in your Telemetry.ex module's start_link/1 function. The handler receives the event name, measurements map, metadata map, and your config. Handlers run synchronously in the emitting process — keep them fast or offload work to a separate process.
